# MTWS 更新日志

## 2026-06-11 · 扫码登录调整为 Nginx 统一 Token 管理

### 登录架构调整
- 扫码登录入口仍保留在各业务软件自己的页面/端口中：MTWS 仍在 MTWS 页面触发登录，OMICS 仍在 OMICS 页面触发登录。
- 登录成功后的 token 不再由某一个业务后端或统一启动器单独持有，改为交给 Nginx 统一管理。
- 不管用户从 MTWS 还是 OMICS 完成扫码登录，Nginx 都会维护同一份当前 token；其他业务请求可复用这份 token，避免多个软件、多个 IP/入口分别登录导致账号互相挤下线。
- MTWS 业务请求需要优先从统一入口/Nginx 管理的登录态中取得 token；本地前端 LocalStorage 仅作为页面展示和兼容缓存，不再作为跨系统 token 的权威来源。

### 中控台调整
- 取消统一中控台/启动器上的扫码登录功能。
- 原登录区域改为说明框，用于展示当前统一登录状态，例如：当前已登录人员、工号/用户标识、登录来源和登录时间。
- 中控台只展示“目前谁登录”，不再负责发起扫码、保存 token 或退出业务登录。

### 与 OMICS 的关系
- OMICS 的登录信息不再放在 OMICS 后端作为统一状态来源。
- OMICS 登录态由前端保存用于页面识别和请求携带；跨 MTWS/OMICS 的共享 token 统一由 Nginx 管理。
- MTWS 与 OMICS 均应通过同源统一入口复用 Nginx 当前 token，减少重复扫码和异地/IP 切换引起的掉线。

## 2026-06-09 · 接入 MTWS+OMICS 统一启动器与 Nginx 同源入口

### 运行架构调整
- MTWS 不再占用对外公共端口 `8000`，改为作为内部 Django 服务监听：`127.0.0.1:8001`。
- 对外访问统一由交付根目录 `MTWS+OMICS` 的启动器管理，通过 Nginx 暴露：
  - MTWS：`http://127.0.0.1:8000/mtws/`
  - OMICS：`http://127.0.0.1:8000/omics/`
- Nginx 只监听本机 `127.0.0.1:8000`，作为同源反向代理入口，不要求目标电脑额外安装系统级 Nginx。

### 启动器集成
- MTWS 由 `MTWS+OMICS\launcher.py` 统一启动、监控和退出。
- 启动器会自动定位 MTWS 目录下的 `mtws_django/manage.py`，并以 `runserver 127.0.0.1:8001` 启动 Django。
- 启动器界面保留 MTWS / OMICS 两个业务服务监控面板；Nginx 作为后台基础设施静默管理，不再单独占用监控栏。
- 点击启动器「退出服务」或托盘右键「退出」时，会统一停止 Nginx、MTWS、OMICS，并清理对应端口残留监听进程。

### 路由兼容
- `http://127.0.0.1:8000/mtws/` 会反向代理到 MTWS 内部服务。
- Nginx 同时兼容 MTWS 旧绝对路径，包括：
  - `/current/`
  - `/test/`
  - `/system/`
  - `/admin/`
  - `/static/`
  - `/media/`
- 根路径 `/mtws/` 会跳转/落到 MTWS 原有 `/current/` 页面逻辑。

### 登录态同步
- MTWS 前端 LocalStorage 已与 OMICS 登录态互通：
  - MTWS 原 key：`mtws_token`、`mtws_userCode`
  - OMICS key：`sf_weather_token`、`sf_userId`
- MTWS 登录成功后会同步写入 OMICS key；读取登录态时也会从 OMICS key 兜底。
- MTWS 退出登录时会同步清理两边 key，减少两个程序重复扫码导致互相挤下线的问题。
